# Задача  - создание прототипа

## Процесс решения
Разработка сервиса началась с создания прототипа по заполнению одного документа. Предварительно был выбран один документ для тестирования работы будущих скриптов и работы с несколькими сценариями взаимодействия.
В процессе создания прототипа решались несколько проблем, с которыми придётся столкнуться по мере продолжения создания новых заполняемых документов.

Во-первых, заполнение документов РКФ состоит не только из ввода текста. На бланках присутствуют и другие элементы, такие как места для проставления галочек, обвода текста на картинках. Такие виды информации также необходимо преобразовать в удобный формат для принятия данных от пользователя.
Во-вторых, при разработке прототипа неоднократно будут вноситься изменения в основной программный код проекта по некоторым причинам: изменения стратегии решения проблем, техническая нереализуемость и прочее.

При первых попытках создать прототип была использована библиотека duckX. Библиотека взаимодействует с файлами docx формата при помощи манипуляций с xml файлами внутри docx. При работе с обычным текстом проблем с заменой текста не возникло. Первая версия прототипа была построена на использовании данной библиотеки. В результате испытаний выяснилось несколько технических особенностей работы:
1. Первая версия маркеров для определения замены выглядела следующим образом - {%словаВнутри%}. Главная задача любого маркера - быть заметным среди множества сочетаний символов и быть уникальным. Такого эффекта обычно добиваются при помощи фигурный скобок или знака процента, так как встретить слово с такими знаками встретить вероятность крайне мала. Однако выяснилось, что по какой-то причине маркера считывались неверно. Вместо одного слова программный код делил маркер на несколько составляющих, из-за чего не происходило нахождение маркера, следовтально, не производилась замена. Было решено сменить формат маркера и убрать любые специальные символы, перейдя на более простую версию - длинные слова, отражающие суть слова, которое должно быть поставлено на место марка. Таким образом **{%breeder%}** превратилось в **PutNameOfTheBreederHere**. Единственный недостаток таких маркеров - это нужно следаить, чтобы в начале маркера не стояло слово, которое нужно менять, иначе произойдёт замена первой части маркера вместо всего.
2. Вторая особенность работы библиотеки заключалась в обработке файла xml. В результате обработки всех строк внутри документа происходили самопроизвольные удаления пробелом в тех местах, где не было ни маркеров, ни каких либо других символом, которые нужно было бы заменить.

Из-за перечисленных проблем было принято решени по замене стратегии обнаружения и замены маркеров. На смену duckX пришёл Python-docx. Идентичная библиотека, но для другого языка порграммирования. Был разработан специальный отдельный скрипт в файле, который будет производить замену. Для доставки информации скрипту было решено попробовать собирать вводимые данные из QLineEdit элементов в JSON. Скрипт читает файл и производит замены без выше упамянутых трудностей с duckX. Кроме скрипта замены был также написан скрипт по копированию файла шаблона, чтобы постоянно не заменять файл вручную.

В ходе разработке пользовательской формы стало очевидно, что она перегружена строками. Разделение на несколько импровизированных "страниц" решило данную проблему. Таким образом был доделан весь шаблон и написан код по его заполнению. Однако схема работы прототипа имеет один ключевой минусов, который невозможно решить в текуйщей реализации. При смене страницы на предыдущую невозможно изменить уже внесённые данные из-за уже отсутствующих маркеров. Для решения данной пробемы нужно разработать схему данных для заполнения всего шаблона. На данный момент JSON файл производит перезапись короткой части вместо собирания всей информации для последующей замены в ходе одного прочтения файла.
