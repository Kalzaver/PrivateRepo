# Задача №2 - выбор типа архитектуры для веб-сервиса

## Процесс решения

Для успешной разработки сервиса нужно выбрать тип его архитектуры.
Выбор лежит между двумя вариантами: монолит и микрос-сервисы.

Основное преимущество микросервисной архитектуры является в способности системы к масштабированию. Микросервисы работают отдельно друг от друга, передавая между собой необходимую информацию. В то же время монолит - система с повышенными связями  между компонентами, которая плохо воспринимает изменения внутри своей работы, обязывая разработчика вносить соотвествующие правки и в остальные части системы. Зачастую изменения в монолите отражаются на всей системе и требуют б**о**льшего контроля для корректного внедрения. На данный момент предполагается, что для проектируемой системы будет предпочтительна архитектура, построенная на микросервисах.

Гибкость микросервисной архитектуры открывает уникальные возможности для технологического разнообразия. Ядро системы, отвечающее за обработку PDF-документов, может быть реализовано на C++ с использованием специализированных библиотек, что обеспечивает максимальную производительность при работе с большими документами. В то же время веб-интерфейс будет использовать более подходящие для этих задач технологии, создавая тем самым оптимальную технологическую экосистему, где каждый инструмент применяется согласно своей задаче. Архитектура микросервисов предоставит возможность по внедрению дополнительного функционала в систему при её дальнейшем развиитии и разработке.

Показательным свойством системы, построенной через управление микросервисами, является устойчивость к отказам, делающее. В традиционных монолитных системах незначительная ошибка в одном модуле может привести к коллапсу всей системы. В микросервисной архитектуре неполадки, например в сервисе уведомлений или временная недоступность сервиса аутентификации, не остановят процесс заполнения документов и система продолжит работу, изолировав проблемный компонент до момента его восстановления, избегая отказа в обслуживании.

Вносить изменения в систему, построенную на микросервисах, проще чем в монолит, так как монолит не предполагает добавления нового функционала за короткое время. Для внедрения необходимо вносить изменения во всю систему и грамонто вносить новый функионал в уже существующий.

## Конечный вывод

В конечном итоге для разрабытываемого веб-сервиса более подходящей видится архитектура типа микросервисов из-за повышенной устойчивости к отказам, упрощённоного внедрения нового функционала.
