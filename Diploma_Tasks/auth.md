# Задача №2 - добавление авторизации

## Процесс решения

Наличие обязательной авторизации перед предоставлением доступа к функционалу проекта - обязательное требование со стороны разработчика. Для дальнейшей, наличие аворизации позволит вести больее подробную статистику использования сервиса и фиксировать конкретные действия за конкретными пользователями. 

Для создания возможности авторизации необходим соответствующий функционал по поиску зарегистрированных пользователей внутри базы данных сервиса. Предварительно создав базу данных SQLite3:

```
CREATE TABLE user ( username VARCHAR(128) PRIMARY KEY, password VARCHAR(128) );
```

в интерфейсе DBeaver вручную создал первого пользователя для разработчика сервиса с логином и паролем admin, admin для проверки работоспособности будущего контроля по доступу к сервису только авторизованным пользователям.

![Изображение интерфейса](//Diploma_Tasks/Images/image.png)

Полноценная регистрация пользователей через использование электронных почт планируется к добавлению, если после проведения полевых испытаний эксперимент по повышению эффективности заполнения документов со стороны РКФ будет признан успешным или выражении одобрения на дальнейшую разработку системы как "альтернативного варианта" заполнения документов с ограниченным доступом только среди участников РКФ. Для этого будет обновлена таблица в базе для указания информации для обратной связи и выдачи персонального идентификатора помимо логина для деперсонализации данных.

Далее при помощи внутренних инструментов, заложенных в Treefrog Prompt создаётся отдельный контроллер и модель для управления пользователем и функционалом по контролю над учётной записью. Сделано это было по нескольким причинам:
1. не перегружать объёмом кода уже существующий контроллер для страниц сервиса
2. соблюсти приницп ООП "один класс - одна задача"

```
 1 $ tspawn usermodel user
 2  created  models/sqlobjects/userobject.h
 3  created  models/user.h
 4  created  models/user.cpp
 5  created  models/models.pro
```

```
1  $ tspawn controller account form login logout
2    created  controllers/accountcontroller.h
3    created  controllers/accountcontroller.cpp
4    created  controllers/controllers.pro
```

Для имения в контроллере сразу заготовок для написания кода и готового в первой строке указывается имя контроллера, имена методов для отображения формы, функционала захода и выхода из учётной записи. В результате создаём пользовательский интерфейс для входа в учётную запись в erb формате.

![Изображение интерфейса](//Diploma_Tasks/Images/экран%20авторизациии.png)

Соответствующий программный код для отслеживания корректиности входа в учётную запись:

```
1  void AccountController::login()
2  {
3      QString username = httpRequest().formItemValue("username");
4      QString password = httpRequest().formItemValue("password");
5  
6      User user = User::authenticate(username, password);
7      if (!user.isNull()) {
8          userLogin(&user);
9          redirect(QUrl("http://localhost:8800/Web/index"));
10     }
11     else {
12         QString message = "Login failed";
13         texport(message);
14         render("form");
15     }
16 }
```

Строки 1-2 отвечают за сравнение получаемых из формы логина и пароля и сравнивают их в базе данных сервиса для проверки факта существования записи. В противном случае пользователя не пропустят:

![Изображение интерфейса](//Diploma_Tasks/Images/провал%20авторизации.png)

Внутренними силами фреймворка при вводе авторизационных данных поле с паролем отображается с заменой видимых символов на звёздочки:

![Изображение интерфейса](//Diploma_Tasks/Images/ввод%20информации%20для%20авторизации.png)

После успешной авторизации пользователь получает доступ к функционалу сервиса и видит свою учётную запись в правом верхнем углу:

```
1 <div class="user-info">
2     <% tfetch(QString, username); %> <p><%= username %></p>
3     <button type="button"><%== linkTo("Выйти", QUrl("http://localhost:8800/account/form")) %></button>
4 </div>
```

![Изображение интерфейса](//Diploma_Tasks/Images/отображение%20аккаунта%20в%20главном%20меню.png)

Контроль факта авторизации пользователя осуществляется при помощи метода preFilter():

```
1 bool WebController::preFilter()
2 {
3     if (!isUserLoggedIn()) {
4         redirect(QUrl("http://localhost:8800/account/form"));
5         return false;
6     }
7     return true;
8 }
```

После проверки переменной **isUserLoggedIn** в строке 3 происходит переадресация пользователя на форму авторизации, в случае, если переменная в значении **false**. Метод preFilter() вызывается при попытке зайти на страницу главного меню в сторке 3:

```
1  void WebController::index()
2  {
3      preFilter();
4  
5      QString username = identityKeyOfLoginUser();
6      User loginUser = User::getByIdentityKey(username);
7  
8      texport(username);
9  
10 	  QString title = "Вы в главном меню"; 
11 	  texport(title);
12     render();
13 }
```
