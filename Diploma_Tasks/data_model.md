# Задача  - создать модель данных

## Процесс решения
Первый прототип по заполнению шаблонного документа имел один существенный недостаток в своей работе. Его логика была устроена таким образом, что замена текста в документе происходила частями.
Такой подход к замене информации в документе существенно усложнял работу с программным кодом и получением конечного результата.
Файл шаблона имеет в себе слова-маркеры. Данные объекты существуют только до тех пор, пока их не заменят на данные, введённые пользователем в пользовательской форме.
Таким образом, логика работы программы оказывается очень плохой, если подумать о возможных проблемах. Замены происходят каждый раз при переходе на новую страницу формы, маркера перезаписываются данными пользователя.
Совсем из фокуса пропал сценарий, когда пользователь захотел изменить старые введённые данные, так как решил их перепроверить и обнаружил ошибку. Текущая логика работы прототипа делает **невозможным** редактирование старых даннных.
При переходе на предыдущую страницу та оказывается пустой, новые данные невозможно перезаписать без обновления шаблона, что будет уничтожать внесённые данные с других страниц. Сплошная головная боль.

Для решения данной фундаментальной проблемы прототипа на помощь приходит понятие "модели данных". Идея простая - создать хранилище для вводимых данных и только в самом конце производить замену данных в файле шаблона. Данная логика работы программы поможет решить все перечисленные ранее проблемы текущего прототипа.

В выбранном фреймворке Qt есть встроенные инструменты для создания собственной модели данных. 
Для их корректной работы необходимо объявить их в конструкторах (строки 4 и 5).

Модель данных создаётся индивидуально для каждого документа в будущем (27 строка). Далее с 29 по 31 строки создаётся единая модель данных для всего документа. Ссылка на созданную модель данных передаётся на следующую страницу для продолжения заполенения (33 - 35 строки).

После проверки наличия модели для записи данных (1 строка) идёт создание части модели данных для первой из несколькиз страниц формы для заполнения документа. Для записи данных в модель используется QDataWidgetMapper (3 строка). С 6 по 15 строки проиходит присодинение lineEdit-ов к мапперу. Такая запись позволит загружать данные из модели по страницам при переходе. 

```
1    MainWindow *second_page = new MainWindow(nullptr, m_model);
2    this->hide();
3    second_page->show();
```

При переходе со страницы на страницу модель будет передаваться и её заполнение продолжится с нового индекса.
Основные трудности, с которыми можно столкнуться при создании такой модели данных:
- недостаточное количество строк для модели в конструкторе модели
- неверный тип данных для модели
- перезапись данных в ходе выполнения кода программы

В ходе разработки новой логики работы прототипа по заполнению копии шаблона выяснилось, что конкретные строки в модели данных нельзя привязать таким же образом к элементам класса QRadioButton как к классу QLineEdit. Для записи данных о нажатии кнопок был переписан старый код прототипа, где нажатая кнопка записывалась как "+" для замены маркера внутри шаблона, вместо галочки в квадратике специального симантического шрифта, вставка которого программным кодом делал файл непригодным для чтения.

Предварительно под каждую радио кнопку готовится отдельная строка, где будет сохранятся плюс, либо минус для отследивания статуса нажатия.
```
QString radioValue28 = m_model->index(0, N).data().toString().trimmed();
```
Таким образом при считывании переменной можно выставить кнопку в нажатое или не нажатое состояние при перезахоже на страницу.
```
1  if (radioValue28 == "+") {
2  ui->radioButton->setChecked(true);
3  } else {
4  ui->radioButton->setChecked(false);
5  }
```
