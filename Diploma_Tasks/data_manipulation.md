# Задача №N - решить проблему манипуляций с данными для заполнения шаблона
## Процесс решения

На данный момент был создан первый прототип решения по заполнению шаблона пользовательскими данными. Форма содержит поля для ввода текста, радио кнопки для проставления галочек. Вводимые данные формируются собираются в json файл, который далее передаётся скрипту на Python для произведения замен. 
Для удобства пользования заполнение формы разделено на несколько страниц, чтобы не перегружать рабочую область пользователя. Данная особенность приводит к другим негативным особенностям работы прототипа, если продолжать его размножать именно в таком виде.
### Основные проблемы протопипа: 
1. пока нет соответствующих проверок вводимых данных: проверка длины ввода, типа данных, пустая ли строка;
2. при переходе со страницы на страницу происходит перезапись json файла;
3. такое парцелирование данных для ввода делает невозможным редактирование уже введённых данных из-за патери маркеров после уже случившейся замены;
4. отключено фиксирование изменений в коде при помощи систем контроля версий;
5. отсуствие тестов для разрабатываемого функционала для проверки коррекности работы проекта;

### Разработке средств по решению проблем созданного прототипа
Сразу можно решить **четвёртую проблему** и подключить систему контроля версий прямо из QtCreator, создав репозиторий командой **git init**. 
**Первая проблема** с проверкой вводимых значений может быть решена несколькими способами:
1. Самый очевидный способ - написать проверки вводимых значений для каждого из QLineEdit элемента через операторы if else. Однако такой способ решения проблемы "в лоб" не является хорошим, так как для  покрытия всех возможных условий, когда вводимые данные будут неверными и не должны быть допущены к записи можеет оказать слишком много, что приведёт к огромному количеству проверок и циклов if else.
```
 1  QString test = ui->lineEdit->text();
 2  if (test.lenth() > 100 || test.lenth == 0)
 3  {
 4      ....
 5  }
 6  else
 7  {
 8      ....
 9  }
```
2. Иной способ - проверка всей вводимой информации только перед формированием файла json. Этот способ также не является идеальным, а в некоторых моментах даже хуже первого. Для достижения корректной проверки снова придётся прибегнуть к большому количеству if else, а также заставить пользователя саостоятельно определять, где он допустил ошибку при вводе, если только не снабдить каждую ошибку специальым сообщением и выводить отдельным окном. По итогу получаем чуть изменённый первый вариант с усложнённой логикой для проверки всей информации уже после его сбора.
3. Следующий способ - использование регулярных выражений для QLineEdit-ов. Это вариант вернее остальных будет справлять со своими задачами по недопущению к записи определённых видов данных. Можно таким образом ограничить приём вводимого значение пользователем на этапе самого ввода. Также такой способ не вызывает проблем с утечкой памяти при его использовании. 

Из трёх пока расммотренных способов будет благоразумнее использовать третий в виду его более гибкой настройки меньщими силами. Базовый пример регулярного выражения выглядит следующим образом:
Для использования приведённой выше конструкцции необходимы дополнительные подключения:
```
1  #include <QRegularExpression>
2  #include <QRegularExpressionValidator>
```
Самое просто регулярное выражение выглядит следующим образом. В первой строке объявлется выражение и создаётся само выражение внутри скобок. 
```
1  QRegularExpression rx("-?\\d{1,3}");
2  QValidator *validator = new QRegularExpressionValidator(rx, this);
3  ui->lineEdit_1->setValidator(validator);
```
Создаём выражение - 1 строка, создаём валидатор для использования написанного выражения - 2 строка, привязываем валидатор к конкретному  QLineEdit элементу на пользовательской форме - 3 строка.
При использовании такого вида проверок не страшны проблемы с утечками памяти, так как валидаторы присоеденены к открытому окну ***this***. После его закрытия память освободится автоматически. Для корректного использования валидаторов и регулярных выражений необходимо правильно расположить их.
Пример:
```
1  void::Widget::on_clicked_switchPage()
2  {
3      QRegularExpression rx("-?\\d{1,3}");
4      QValidator *validator = new QRegularExpressionValidator(rx, this);
5      ui->lineEdit_1->setValidator(validator); 
6  }
```
Написанное внутри скобок регулярное выражение не будет давать возможность вводить ничего кроме цифр и то не более трёх штук. Выражение несомненно можно изменить, что будет сделано в будущем для всех. Данный валидатор не будет работать так как нужно нам. Таким образом невозможно обеспечить проверку в момент ввода информации в QLineEdit. Причина данной проблему в том, что код валидатора, привязки к lineEdit_1 и объявления регулярного выражения находяься внутри кода, выполняемого только по нажатию кнопки. Следовательно, коды подобной конструкции нельзя располагать внутри кнопок. В нашем случае необхожимо расположить код внутри конструктора, как в примере ниже.

```
1  Widget::Widget(QWidget *parent)
2      : QWidget(parent)
3      , ui(new Ui::Widget)
4  {
5      ui->setupUi(this);
6      connect(ui->pushButton_forward1, &QPushButton::clicked, this, &Widget::on_clicked_switchPage);
7
8      QRegularExpression rx("-?\\d{1,3}");
9      QValidator *validator = new QRegularExpressionValidator(rx, this);
10     ui->lineEdit_1->setValidator(validator);
11 }
```
Таким образом можно сделать лишь несколько спецциальный регулярных выражений для проверки в момент ввода текста в QLineEdit и ссылаться на уже созданные валидаторы для всех элементов ввода на пользовательской форме. В конечном итоге получается короткий и читаемый програмный код. 
```
1     QRegularExpression line1(R"(^[\p{L} ]{0,90}$)");
2     QValidator *valid1 = new QRegularExpressionValidator(line1, this);
3     ui->lineEdit_1->setValidator(valid1);
4     ui->lineEdit_2->setValidator(valid1);
5     ui->lineEdit_3->setValidator(valid1);
6
7     QRegularExpression line4("-?\\d{1,7}");
8     QValidator *valid4 = new QRegularExpressionValidator(line4, this);
9     ui->lineEdit_4->setValidator(valid4);
10
11    QRegularExpression line5(R"(^[\p{L} ]{0,45}$)");
12    QValidator *valid5 = new QRegularExpressionValidator(line5, this);
13    ui->lineEdit_5->setValidator(valid5);
14    ui->lineEdit_6->setValidator(valid5);
15    ui->lineEdit_7->setValidator(valid5);
16    ui->lineEdit_8->setValidator(valid5);
17    ui->lineEdit_9->setValidator(valid5);
18    ui->lineEdit_10->setValidator(valid5);
```
Что фильтрует каждое из регулярных выражений:
1 строка - допускаются к набору строки из букв до 90 символов.
7 строка - допускается к набору только цифры и не более 7 штук.
11 строка - допускаются к набору строки из букв до 45 символов.

Далее для покрытия других страниц будут созданы выражения для других длин строк, проверки на наличие строк и смена работы по формированию файла для загрузки информации в скрипт по замене.
